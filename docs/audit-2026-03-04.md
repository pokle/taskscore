# TaskScore Codebase Audit — 2026-03-04

## Codebase Overview

TaskScore is a client-side web application for analyzing hanggliding/paragliding competition track logs (IGC files). The codebase is organized as a monorepo with three main packages:

- **`web/engine`** — Core computation library: IGC parsing, XCTask parsing, event detection, circle/thermal detection, turnpoint sequencing, task optimization, glide analysis, unit formatting, and waypoint matching.
- **`web/frontend`** — Vite-based SPA: map rendering (MapBox GL + Leaflet), analysis panel UI, file loading, IndexedDB storage, configuration, and AirScore integration.
- **`web/workers/airscore-api`** — Cloudflare Worker: caching proxy for the AirScore competition management API.

The architecture is well-designed with clean separation of concerns. The engine is pure computation with no DOM/browser dependencies, the frontend handles all presentation, and the worker handles external API proxying. Test coverage is good for the engine but absent for the worker.

## Bug Findings

### Critical / High Severity

#### ~~BUG-01: XSS via unescaped HTML interpolation from external data~~ [Fixed](https://github.com/pokle/taskscore/pull/71)

**Files:** `web/frontend/src/analysis/analysis-panel.ts` (multiple locations), `web/frontend/src/analysis/leaflet-provider.ts` (DivIcon HTML), `web/frontend/src/analysis/mapbox-provider.ts` (marker HTML)

Multiple places build HTML strings using `innerHTML` with data sourced from external IGC files, XCTask JSON, or the AirScore API — without HTML escaping:

- **Pilot name** from IGC headers → `setFlightInfo()` renders `info.pilot` via `innerHTML` (~line 1348). A malicious IGC file with `<img src=x onerror=alert(1)>` as pilot name executes JavaScript.
- **Event descriptions** → rendered via `innerHTML` in the events list (~line 643).
- **Waypoint names** from task files → `tp.waypoint.name` rendered unescaped in the task panel (~line 1118), turnpoint labels on the map, and Leaflet DivIcon HTML.
- **Glider type**, date, and other IGC header fields flow through `innerHTML` in the flight info section.

An `escapeHtml()` function exists in `storage-menu.ts` but is not used in the analysis panel or map providers.

**Impact:** Any user who opens a crafted IGC or XCTask file could have arbitrary JavaScript executed in their browser.

#### ~~BUG-02: `Math.max(...array)` stack overflow on large tracks~~ [Fixed](https://github.com/pokle/taskscore/pull/70)

**File:** `web/frontend/src/analysis/main.ts:945`

```typescript
const maxAlt = Math.max(...state.fixes.map(f => f.gnssAltitude));
```

Spreading all fix altitudes as function arguments exceeds the call stack for long flights. A typical 5-hour flight at 1-second intervals produces ~18,000 fixes. The argument limit varies by engine but is typically 65,536 in V8 — however some environments crash at lower counts, and a 10-hour flight at 1-second intervals would hit ~36,000 fixes which is borderline. Should use a loop-based max.

#### ~~BUG-03: Fix index offset not applied to altitude/vario/turnpoint events~~ [Fixed](https://github.com/pokle/taskscore/pull/73)

**File:** `web/engine/src/event-detector.ts:926-932`

The main `detectFlightEvents` function slices `fixes` from the takeoff index to create `flightFixes`, then applies an `indexOffset` to thermal/glide events so their segment indices map back to the original array. However, `detectAltitudeExtremes`, `detectVarioExtremes`, and `detectTurnpointEvents` are called with `flightFixes` but their results are pushed into `allEvents` **without** adjusting `fixIndex` in their detail objects. Any downstream code that uses `details.fixIndex` to index into the original `fixes` array will access the wrong position.

```typescript
// Lines 926-932: fixIndex in details will be relative to flightFixes, not fixes
allEvents.push(...detectAltitudeExtremes(flightFixes));
allEvents.push(...detectVarioExtremes(flightFixes));
if (task) {
  allEvents.push(...detectTurnpointEvents(flightFixes, task));
}
```

#### ~~BUG-04: IGC parser does not handle midnight rollover~~ [Fixed](https://github.com/pokle/taskscore/pull/72)

**File:** `web/engine/src/igc-parser.ts:93-101`

The `parseTime` function sets hours/minutes/seconds on a base date but does not handle flights crossing midnight UTC. Fixes after midnight will have timestamps that appear to go backwards (e.g., 23:59 → 00:01 on the same date). This produces negative time differences and breaks event detection, duration calculations, and glide speed computations. This is a real-world scenario for flights in certain time zones (e.g., evening flights in the Pacific region).

#### ~~BUG-05: SSS crossing direction not validated~~ [Fixed](https://github.com/pokle/taskscore/pull/76)

**File:** `web/engine/src/turnpoint-sequence.ts:491-493`

When detecting SSS (Speed Section Start) crossings, the code accepts any crossing direction. However, `task.sss.direction` specifies the required direction (typically `'EXIT'` for paragliding races — pilots must fly out of the SSS cylinder). A pilot who enters but never properly exits the SSS cylinder could be incorrectly scored as having started. The `CylinderCrossing.direction` field exists on the result objects but is never checked against `task.sss.direction`.

#### ~~BUG-06: IGC validation logic inverted in worker track handler~~ [Fixed](https://github.com/pokle/taskscore/pull/75)

**File:** `web/workers/airscore-api/src/handlers/track.ts:34-37`

```typescript
if (!content.startsWith('A') && !content.includes('HFDTE')) {
    throw new Error('Response does not appear to be a valid IGC file');
}
```

Uses `&&` (AND) instead of `||` (OR). This means invalid content is only rejected if it fails **both** checks. An HTML error page starting with `<A` (like `<A href=...`) would pass validation because it starts with 'A'. The condition should reject if **either** check fails.

### Medium Severity

#### ~~BUG-07: `preferences-changed` event causes unnecessary re-detection on map move~~ [Fixed](https://github.com/pokle/taskscore/pull/77)

**File:** `web/frontend/src/analysis/main.ts:430` + `config.ts:77-93`

When the map saves its position (debounced at 5 seconds), `config.setMapLocation()` calls `setPreferences()`, which dispatches `taskscore:preferences-changed`. The listener at line 430 calls `redetectEvents()` on every preference change — not just unit changes. This causes all flight events to be re-detected, the task to be re-rendered on the map, and the analysis panel to re-render every time the map saves its position. This is both wasteful and can cause visible UI flicker.

#### BUG-08: Status message auto-hide timer not cancelled

**File:** `web/frontend/src/analysis/main.ts:994-999`

The 3-second `setTimeout` for success messages is never cleared. If `showStatus('Loading...', 'info')` is called, then `showStatus('Done!', 'success')` sets a 3-second hide timer, then `showStatus('Error!', 'error')` is shown — the error message will be hidden by the stale timer after just 2 seconds.

#### BUG-09: Division by zero in `calculateGlidePositions`

**File:** `web/engine/src/glide-speed.ts:77`

```typescript
const t = Math.max(0, Math.min(1, 1 - (overshoot / segmentDistance)));
```

When `segmentDistance` is 0 (consecutive fixes at the same GPS position), this produces `NaN` from division by zero. `Math.max(0, NaN)` returns `NaN`, so all interpolated values (lat, lon, time, altitude) become `NaN`. This corrupts the positions array and all downstream glide marker calculations.

#### BUG-10: `task-line-arrows` layer not included in task visibility toggle

**File:** `web/frontend/src/analysis/mapbox-provider.ts:987-998`

`setTaskVisibility` toggles visibility for `task-line`, `task-cylinders-fill`, `task-cylinders-stroke`, `task-points`, `task-labels`, and `task-segment-labels`, but does **not** include `task-line-arrows`. When task visibility is toggled off, the arrow markers along the task line remain visible on the map.

#### BUG-11: `highlight-segment` layer not restored after track visibility toggle

**File:** `web/frontend/src/analysis/mapbox-provider.ts:1002-1036`

When `setTrackVisibility(false)` is called, the `highlight-segment` layer visibility is set to `'none'`. When `setTrackVisibility(true)` is called, `updateTrackRendering()` only restores `track-line` and `track-line-outline`. The `highlight-segment` layer stays hidden. Selecting events after re-enabling track visibility won't show the highlighted segment on the map.

#### BUG-12: Best progress calculation overestimates flown distance

**File:** `web/engine/src/turnpoint-sequence.ts:381-414`

The `computeBestProgress` function measures remaining distance as straight-line to the goal cylinder. Per CIVL GAP rules, remaining distance should be the shortest optimized path through any un-reached intermediate turnpoints to goal. If a pilot tags SSS and TP1 but misses TP2 and TP3, the flown distance credit will be overestimated because the remaining distance doesn't account for the required path through TP2 and TP3.

#### BUG-13: Golden section search can miss global minimum for overlapping cylinders

**File:** `web/engine/src/task-optimizer.ts:66-92`

The golden section search assumes the cost function (total distance through a point on a turnpoint circle) is unimodal on [0, 2π]. When the previous or next turnpoint center is inside the current turnpoint's cylinder (overlapping cylinders with large radii), the cost function can have two local minima. The search will converge to one, potentially missing the true global minimum and computing a suboptimal task distance.

#### BUG-14: Cache stores raw data but transformation runs on every cache hit

**File:** `web/workers/airscore-api/src/handlers/task.ts:94-101`

The AirScore raw response is cached in KV, but `transformResponse()` runs on every request including cache hits. The transformation includes HTML parsing for pilot data. Should cache the transformed result instead.

#### BUG-15: `mapWaypointType` handles `'takeoff'` (not in type union) but misses `'start'`

**File:** `web/workers/airscore-api/src/transforms/task.ts:21-32`

The `AirScoreWaypoint` type defines `tawType: 'start' | 'speed' | 'waypoint' | 'endspeed' | 'goal'`. The mapping function handles `'takeoff'` (which isn't in the union!) but doesn't handle `'start'` or `'goal'`. The `'start'` type likely refers to the start turnpoint and should probably map to something appropriate.

#### BUG-16: IndexedDB `clearAllTasks`/`clearAllTracks` resolve too early

**File:** `web/frontend/src/analysis/storage.ts:384-413`

Both clear operations resolve on `request.onsuccess` instead of `tx.oncomplete`. The `store.clear()` request success does not guarantee the transaction has committed. If code immediately reads from storage after clearing, it may see stale data.

#### BUG-17: `parseAirScoreUrl` doesn't validate parsed integers

**File:** `web/frontend/src/analysis/main.ts:859-880`

`parseInt(comPk, 10)` doesn't check for `NaN`. If `comPk` is "abc", the result `NaN` is passed to `fetchAirScoreTask` and embedded in URL query strings. The task handler in the worker validates with `isNaN` but this creates an unnecessary round trip.

### Low Severity

#### BUG-18: `normalizeBearingDelta` asymmetric range

**File:** `web/engine/src/circle-detector.ts:96`

```typescript
if (d <= -180) d += 360;  // Note: <= not <
```

The range is `(-180, 180]` not `[-180, 180]`. A delta of exactly -180° maps to +180°. This creates a subtle positive bias in cumulative heading that could occasionally misclassify turn direction at the boundary.

#### BUG-19: `getBoundingBox` returns (0,0) for empty arrays

**File:** `web/engine/src/geo.ts:131`

Returns `{minLat: 0, maxLat: 0, minLon: 0, maxLon: 0}` for empty arrays. This is indistinguishable from "all points at (0,0)" and could center the map on the Gulf of Guinea.

#### BUG-20: Glide description shows "Infinity" for flat glides

**File:** `web/engine/src/event-detector.ts:897`

When `altLoss <= 0`, `glideRatio` is `Infinity`. The description string `Glide start (L/D ${glide.glideRatio.toFixed(0)})` renders as "Glide start (L/D Infinity)".

#### BUG-21: `filterEventsByBounds` doesn't handle anti-meridian

**File:** `web/engine/src/event-detector.ts:982-997`

The bounds filter assumes `west < east`, which is false when the viewport crosses the 180th meridian. Events near the dateline would be incorrectly filtered out.

#### BUG-22: `findWaypointByName` substring matching is too loose

**File:** `web/engine/src/waypoints.ts:109-113`

```typescript
const contained = waypoints.find(wp =>
  upperName.includes(wp.name.toUpperCase()) || ...
);
```

Searching for "TURNPOINT ORANGE" would match a waypoint named "RANGE". Single/two-letter waypoint names would match almost anything.

#### BUG-23: `updateUrlParam` leaves trailing `?` when all params removed

**File:** `web/frontend/src/analysis/main.ts:1005-1014`

When all params are deleted, `params.toString()` returns empty string but the URL is constructed as `${pathname}?`, leaving a trailing `?`.

#### BUG-24: `saveLocationTimer` not cleared on provider `destroy()`

**File:** `web/frontend/src/analysis/mapbox-provider.ts` and `leaflet-provider.ts`

Both providers set up a debounced timer for persisting map location. The `destroy()` method doesn't clear pending timers, which could fire after the map is removed and throw errors.

#### BUG-25: Unit conversion factors slightly imprecise

**File:** `web/engine/src/units.ts:29-44`

- `mph` factor 2.237 should be 2.23694
- `ft` factor 3.281 should be 3.28084
- `knots` factor 1.944 should be 1.94384

While used for display only, accumulated error could affect rounding at boundaries.

#### BUG-26: Wildcard CORS on worker API

**File:** `web/workers/airscore-api/src/index.ts:18`

`Access-Control-Allow-Origin: *` allows any website to use this worker as an open proxy to AirScore. Could be used to abuse the upstream API.

#### BUG-27: Worker task handler accepts negative integer IDs

**File:** `web/workers/airscore-api/src/handlers/task.ts:78-88`

`parseInt` accepts negative values. The track handler validates with a regex (`/^\d+$/`) but the task handler does not. Negative IDs would be passed to the upstream API.

#### BUG-28: `parseFloat` without NaN guard on quality/goalPenalty

**File:** `web/workers/airscore-api/src/transforms/task.ts:133,144`

`parseFloat(task.quality)` and `parseFloat(formula.goal_penalty)` return `NaN` for empty strings. These `NaN` values propagate to the client as numeric fields.

#### BUG-29: Non-deterministic tests using `Math.random()`

**File:** `web/engine/tests/circle-detector.test.ts:510-529`

Two tests use `Math.random()` for fix intervals and noise, making them flaky. Should use a seeded PRNG or deterministic values.

#### BUG-30: `ResizeObserver` leak in HUD draggable

**File:** `web/frontend/src/analysis/map-provider-shared.ts:612`

`new ResizeObserver().observe(container)` is never disconnected. If HUD is recreated, old observers leak.

#### BUG-31: `evaluateTakeoffCriteria` latent out-of-bounds access

**File:** `web/engine/src/event-detector.ts:534`

`calculateGroundSpeed(fixes[index - 1], fixes[index])` would crash if `index === 0`. Currently safe because `detectTakeoff` starts the loop at `i = 1`, but there's no guard in the function itself.

#### BUG-32: `avgSinkRate` sign convention inconsistency

**File:** `web/engine/src/segment-extractors.ts:192`

`avgSinkRate` is a positive number for descent, opposite the conventional negative sign for sink rate. Could confuse consumers.

#### BUG-33: `detectLanding` misses landing for very short flights

**File:** `web/engine/src/event-detector.ts:621`

The loop starts at `fixes.length - 2` and stops at `config.landingTimeWindow` (30). Flights with fewer than ~32 fixes will never detect a landing.

#### BUG-34: IGC task `numTurnpoints` parsing edge case

**File:** `web/engine/src/igc-parser.ts:286`

With exactly 3 task C-record points, `numTurnpoints = Math.max(0, 3 - 4) = 0`. The code assigns takeoff, start, and landing but never assigns finish — leaving the task in an inconsistent state.

## Test Coverage Gaps

1. **No tests for the worker** — The entire `web/workers/airscore-api` package has zero tests. The transform functions, handlers, cache logic, and routing are all untested.
2. **No midnight-crossing IGC test** — The parser is not tested with B-records that cross midnight UTC.
3. **No malformed B-record test** — Real IGC files from flight recorders frequently have garbled or truncated lines.
4. **No `parseXCTask` error handling test** — Invalid JSON, missing fields, and invalid coordinates are not tested.
5. **No `filterEventsByBounds` anti-meridian test** — Bounding boxes crossing the 180th meridian are not tested.
6. **No XSS/injection test** — No tests verify that user-supplied data is properly escaped before HTML rendering.

## Architecture Observations (Non-Bug)

1. **Clean engine/frontend separation** — The engine package has zero browser dependencies, making it highly testable and potentially reusable in Node.js CLI tools.
2. **Good use of typed interfaces** — TypeScript interfaces are well-defined throughout, with clear separation between internal and public types.
3. **Duplicated type definitions** — `UnitPreferences`, `SpeedUnit`, etc. are defined in both `web/engine/src/units.ts` and `web/frontend/src/analysis/config.ts`. Consider re-exporting from a single source.
4. **Event detection pipeline is solid** — The thermal/glide detection, circle detection, and turnpoint sequencing are well-implemented with good test coverage and clear algorithm documentation.
5. **Map provider abstraction works well** — The `MapProvider` interface cleanly abstracts over MapBox GL and Leaflet, with shared code properly extracted to `map-provider-shared.ts`.
